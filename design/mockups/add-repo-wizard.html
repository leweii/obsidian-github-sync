Add Repo Wizard — expanded flow

Page 3 of the design plan. Replaces the current single-screen AddSubmoduleModal with a 4-step wizard that handles the unresolved design gap from the brainstorm: what to do when the target folder already contains files. Three explicit branches — adopt, reject, overwrite — with the destructive one gated by type-to-confirm.

  1. Step 1 — Source — vault/folder choice + URL probe + branch
  2. Step 2a — Mount (clean) — happy path, target folder is empty/missing
  3. Step 2b — Mount (COLLISION) — folder already has files, three-way branch decision
  4. Step 3 — Cloning — progress bar with current stage + log tail
  5. Step 4 — Done — success summary + next action
  6. Error variant — network or auth failure with retry path
Step 1

Source — URL + branch + scope

User picks scope (entire vault vs. a folder), pastes a GitHub URL, and the wizard probes GitHub for default branch + reachability inline. The "Folder" card auto-fills the path field from the repo name — fewer keystrokes than the current AddSubmoduleModal.

Step 2b — KEY DECISION

Mount — folder already has files

This is the design gap from the vault brainstorm: the target folder already contains user files. Three explicit branches with safety-graded visual treatment: Adopt (safe, recommended) / Pick another path (safe) / Overwrite (destructive — gated by type-to-confirm). Existing file list is shown as a preview so the user knows what they're choosing about.

Adopt — merge existing files into the repo Recommended · Safe

Pick another path Safe

Step 3

Cloning — progress

Five sub-stages: resolving HEAD → receiving objects → resolving deltas → checking out → configuring submodule. Each shows in clone-stage with a token-count detail underneath. Live log tail (last 4 lines) gives engineers comfort that something is happening; non-technical users can ignore it. Cancel works — it aborts the clone and goes back to Step 2.

Error variant

Cloning failed — network or auth error

If the clone fails partway, the wizard shows an error state instead of jumping back. User can Retry (same params), Back (re-check URL/branch), or paste the raw error when reporting. Stays in the wizard so the user doesn't lose Step 1 state.
